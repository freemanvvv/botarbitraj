# Construction AI Copilot для Узбекистана
## Реалистичное многофазное ТЗ под MacBook M3 (20 GB Unified Memory)

> Версия документа: рабочая, фазовая.
> Принцип: сначала то, что **реально работает** на этом железе и даёт ценность, потом — экспериментальное.
> Как использовать: можно собирать вручную по фазам **или** скармливать локальному агенту (LM Studio + Cline/Roo Code) по одной фазе за раз. Не подавать весь документ как одну задачу — на 8–14B модели это даст поверхностный результат.

---

## 0. Честные ограничения железа (читать первым)

MacBook M3, 20 GB unified memory — это **вход в локальный AI**, а не платформа «всё сразу». Реальный бюджет памяти:

| Потребитель | Память |
|---|---|
| macOS + браузер + редактор | ~4–6 GB |
| Доступно под ML и сервисы | **~14–15 GB** |
| LM Studio рекомендует держать под GPU offload | не более ~75% доступного |

Из этого следуют жёсткие правила проекта:

1. **Одна LLM в памяти за раз.** LM Studio переключает модели, но держать 14B и 8B одновременно нельзя.
2. **14B и полный Docker-стек вместе не работают.** Qwen3-14B Q4 (~9 GB весов + KV-кэш контекста) + Postgres + ChromaDB + Open WebUI + ERP + OCR = выход за память и swap. Выбираем одно.
3. **Контекст 32k на 14B — дорогой.** KV-кэш съедает несколько GB. На M3 реалистично держать 14B на контексте **8–16k**, а 32k — только при выгруженном Docker и коротких сессиях.
4. **Vision локально недоступен.** На 20 GB полноценной модели «понимания чертежей» нет. Анализ чертежей/сканов → только OCR (текст), а не понимание геометрии.

**Ориентир по весам (Q4_K_M):** Qwen3-8B ≈ 4.5–5 GB · Llama-3.1-8B-Uz ≈ 4.5–5 GB · Qwen3-14B ≈ 8.5–9 GB (+ контекст сверху).

## 1. Принципы проекта

**Разделение ответственности — ключевое.** LLM на 8–14B силён в одном и слаб в другом:

- ✅ **Хорошо:** поиск по нормативам (RAG), объяснение требований, генерация текстовых документов (ТЗ, пояснительные записки), структурирование смет/BOQ в таблицы, перевод и локализация RU↔UZ, написание Python-скриптов под IfcOpenShell.
- ⚠️ **Только как черновик под проверку инженера:** инженерные рекомендации, прикидка стоимости, генерация структуры IFC.
- ❌ **Не доверять LLM:** расчёт несущих конструкций и нагрузок, поверочные расчёты на безопасность, геометрически осмысленное проектирование зданий, «понимание» чертежа по картинке.

**Правило безопасности:** всё, что касается прочности/безопасности конструкций, LLM **не считает**. Считают детерминированные инструменты и/или инженер. LLM только готовит текст и черновики.

**Двуязычность.** RU и UZ обеспечиваются на двух уровнях: (а) выбором мультиязычной embedding-модели для RAG, (б) маршрутизацией узбекских запросов на Llama-3.1-8B-Instruct-Uz.

## 2. Базовая инфраструктура (общая для всех фаз)

### 2.1 LM Studio

- Установить LM Studio, включить **Apple Metal** acceleration.
- GPU Offload: выставить максимально возможный, но оставить запас памяти под систему (не 100%).
- Quantization: **Q4_K_M** для всех трёх моделей (лучший баланс качество/память на 20 GB).
- Включить **OpenAI-compatible API** на `http://localhost:1234/v1`.
- Проверить эндпоинты: `GET /models`, `POST /chat/completions`, `POST /embeddings`.

### 2.2 Модели и параметры

| Модель | Назначение | Temp | Top P | Контекст (реалистичный на M3) |
|---|---|---|---|---|
| `behbudiy/Llama-3.1-8B-Instruct-Uz` | узбекский, КМК, диалог с заказчиком, ТЗ, локализация | 0.4 | 0.95 | 16384 |
| `Qwen3-8B` | быстрые консультации, ответы по документации, простые задачи | 0.3 | 0.9 | 16384 |
| `Qwen3-14B Q4_K_M` | сметы, инженерный текст, сложные агентные задачи, генерация скриптов | 0.3 | 0.9 | **8192–16384** (32k только без Docker) |

> Hermes исключён по требованию.

### 2.3 Embedding-модель (критично для качества RAG на RU/UZ)

Отдельно от LLM нужна мультиязычная embedding-модель. Рекомендация: **`bge-m3`** или `multilingual-e5-large` — обе хорошо покрывают русский и приемлемо узбекский. Запускать через LM Studio (`/embeddings`) или локально. Это сильнее влияет на качество поиска по нормативам, чем выбор самой LLM.

### 2.4 Маршрутизация (Construction AI Copilot — главный агент)

Простая логика выбора модели:

- Запрос на узбекском (детект языка) → **Llama-3.1-8B-Uz**
- Простой вопрос / быстрый ответ по документации → **Qwen3-8B**
- Сметы, инженерный текст, агентные задачи, генерация скриптов → **Qwen3-14B**

Реализация роутера — тонкий Python-слой поверх LM Studio API (детект языка + классификатор сложности по ключевым словам/длине). Не усложнять на старте.

## 3. ФАЗА 1 — RAG-ассистент по нормативам + генерация документов
### (Это ядро. Реально работает на M3. С неё начинаем.)

**Цель:** двуязычный ассистент, который отвечает по КМК Узбекистана / СНиП / ГОСТ с ссылками на источник и генерирует текстовые документы (ТЗ, пояснительные записки) в PDF/DOCX.

### Компоненты
- LM Studio: Qwen3-8B + Llama-3.1-8B-Uz (14B на этой фазе не обязателен).
- **ChromaDB** — запускать как встроенную Python-библиотеку (НЕ отдельным Docker-контейнером), чтобы экономить память.
- **LlamaIndex** — оркестрация RAG (загрузка, чанкинг, retrieval, сборка промпта).
- Embeddings: `bge-m3` через LM Studio.
- **PyPDF / pdfplumber** — парсинг текстовых PDF нормативов.
- Генерация документов: Markdown → **DOCX** (`python-docx`) и **PDF** (через `weasyprint` или `markdown-pdf`).

### Что делает
- Индексирует загруженные нормативы (КМК, СНиП, ГОСТ) в локальную векторную базу.
- Отвечает на вопросы по нормам **с цитированием пункта/документа**.
- Генерирует ТЗ и пояснительные записки по шаблонам, экспортирует в PDF/DOCX.
- Переключается RU/UZ.

### Демо-сценарии, закрываемые Фазой 1
- **№2:** «Проверь лестницу с высотой ступени 210 мм на соответствие КМК» → поиск пункта норматива + вывод соответствия с цитатой. *(LLM находит и цитирует норму; вердикт даёт на основе найденного текста, а не «из головы».)*
- **№5:** «Сформируй пояснительную записку и экспортируй в PDF» → генерация текста + экспорт.

### Критерии приёмки Фазы 1
- На вопрос по норме ответ содержит ссылку на конкретный документ/пункт, а не выдумку.
- ТЗ и пояснительная записка корректно выгружаются в PDF и DOCX.
- Узбекские запросы обрабатываются Llama-Uz без потери смысла.
- Всё работает в пределах памяти без swap (контроль через Activity Monitor).

### Ограничения Фазы 1
- Качество ответа ограничено качеством загруженной базы: **мусорные сканы без OCR не индексируются** (OCR — отдельная мини-задача, см. ниже).
- 8B модель иногда «уверенно ошибается» — поэтому обязательна привязка к цитате, чтобы пользователь мог проверить.

## 4. ФАЗА 2 — Сметы и BOQ
### (Структурированная задача — сильная сторона LLM. Включаем 14B.)

**Цель:** генерация смет и ведомостей объёмов работ (BOQ) с расчётом стоимости по локальным расценкам.

### Компоненты
- LM Studio: **Qwen3-14B** (на этой фазе Docker-стек держим выгруженным).
- База расценок: локальная **SQLite** или CSV (материалы, работы, цены по Ташкенту/регионам).
- Экспорт: **XLSX** (`openpyxl`) для смет/BOQ + PDF-отчёт.
- Опционально: SQLite MCP-сервер, чтобы агент сам доставал расценки.

### Что делает
- По описанию объекта формирует ведомость работ и материалов.
- Подтягивает цены из локальной базы, считает стоимость (арифметику делает **код**, не LLM — LLM только структурирует состав работ).
- Выгружает смету/BOQ в XLSX и PDF.

### Демо-сценарий
- **№3:** «Рассчитай стоимость коробки дома в Ташкенте» → состав работ от LLM + расценки из базы + расчёт в коде + выгрузка.

### Критерии приёмки Фазы 2
- Итоговые суммы считаются кодом по базе расценок, а не «придумываются» моделью.
- BOQ выгружается в XLSX с разбивкой по разделам.
- Цены вынесены в редактируемую базу (легко обновлять под рынок).

### Ограничения Фазы 2
- Точность сметы = точность базы расценок. LLM не знает актуальных цен — их даёшь ты.
- Сложные сметы требуют ручной выверки состава работ.

## 5. ФАЗА 3 — BIM / IFC (экспериментально, с явными оговорками)
### (Самая слабая для локальной LLM зона. Делаем как генерацию скриптов-черновиков.)

**Цель:** не «ИИ проектирует здание», а **«ИИ пишет Python-скрипты под IfcOpenShell для создания простых элементов и базовой геометрии, которые проверяет инженер»**.

### Компоненты
- LM Studio: **Qwen3-14B** (генерация кода).
- **IfcOpenShell** — чтение/создание/редактирование IFC через Python API.
- **Bonsai (Blender BIM)** — визуальная проверка и ручная доводка IFC.
- Экспорт планов: **SVG** (генерация по параметрам), **DXF** (`ezdxf`), PNG/PDF.
- Опционально: MCP4IFC, Python MCP, Filesystem MCP для агентного выполнения.

### Что делает (реалистично)
- Генерирует Python-скрипт, создающий **простые** IFC-элементы (стены, перекрытия, проёмы по заданным размерам).
- Строит схематичный **SVG-план** по параметрам (сетка осей, контур, помещения как прямоугольники).
- Открывает результат в Bonsai для проверки и правки человеком.

### Демо-сценарии (с пометкой «черновик»)
- **№1:** «Двухэтажный дом 180 м² с гаражом на 2 машины» → LLM выдаёт ТЗ + примерную планировку (текст/SVG) + скрипт-заготовку IFC. **IFC будет синтаксически валидным, но геометрию и осмысленность проверяет архитектор.**
- **№4:** «SVG-план первого этажа» → схематичный план по параметрам.

### Жёсткие ограничения Фазы 3 (обязательно к пониманию)
- LLM **не спроектирует** дом сам. Она генерирует код и схемы по твоим явным размерам и составу помещений.
- Сгенерированный IFC = **черновик-заготовка**, не проектная документация.
- Никаких автоматических «цифровых двойников» и расчётов нагрузок здесь нет — это вне возможностей локальной 14B.

## 6. Вынесено за рамки / отложено (и почему)

| Что | Почему отложено | Альтернатива |
|---|---|---|
| Анализ чертежей/фото по изображению | Vision-модель не помещается в 20 GB | OCR текста (Tesseract/PaddleOCR) для извлечения надписей; «понимание» — облако в будущем |
| Расчёт нагрузок и поверочные расчёты | Вопрос безопасности; LLM ненадёжна | Специализированный расчётный софт + инженер |
| Полный Docker-стек (ERP + Postgres + OCR + MCP) одновременно с 14B | Не влезает в память | Запускать сервисы по одному / на отдельном сервере (см. масштабирование) |
| OpenConstructionERP, 4D/5D BIM | Тяжёлый стек, отдельный проект | Поднимать на выделенной машине после Фазы 2 |

**OCR как отдельная мини-задача:** можно добавить в любую фазу как изолированный скрипт (Tesseract или PaddleOCR) для превращения сканов нормативов в текст перед индексацией в RAG. Не держать как постоянный сервис.

## 8. Масштабирование (когда упрёшься в M3)

Проект изначально проектируется переносимым: LM Studio API совместим с OpenAI, поэтому смена «движка» не ломает код.

- **RTX 4090 (24 GB VRAM):** 14B без компромиссов + полный контекст 32k; можно крутить Docker-стек параллельно.
- **Mac Studio Ultra (128–192 GB):** 70B-модели, реально лучшее качество инженерного текста; всё локально и одновременно.
- **2× V100 32 GB / выделенный Linux-сервер:** продакшн-развёртывание полного Docker-стека (ERP, Postgres, OCR, MCP, Open WebUI) + крупная модель; сюда же уезжает vision-анализ чертежей.

Стратегия: на M3 доводим Фазы 1–2 и черновую Фазу 3, затем переносим инференс на сервер, оставляя архитектуру и код без изменений.
